## Relation Table

### terminology

- relation: 표 (table)
- record: 줄 (row)
- field: 줄 (column)
- primary key: unique해야함
- dimension - categorical data이면서 쪼개서 분석할 수 있는 단위 (ex. gender)

### independency

column/row 단이 independent한지 아닌지를 생각해야
ex. employee table는 각 row가 independent, purchase table은 각 row가 dependent

예를 들어, regression은 각 variable이 independent한 것이 전제인데, 주식투자는 time serial한 경우이고 각 시기별 price가 dependent하니 회귀분석을 돌리면 안됨

각 데이터가 independent한지 아닌지 확인할 수 있는 방법이 따로 있음

### relation

DB를 만들때, 오래 보관해야하는 것은 최대한 분할해두고, 자주 쓰는 것은 분석을 위해 한꺼번에 할 수 있게

다른 테이블의 primary key는 내게 오면 foreign key가 됨

```SQL
SELECT CUSTOMER,고객이름, CUSTOMER.성별, PURCHASE.구매금액
FROM CUSTOMER, PURCHASE
```

example 3 from p.12

```SQL

FROM CUSTOMER, PURCHASE, PRODUCT
WHERE CUSTOMER.고객ID = PURCHASE.고객ID
    AND PRODUCT.상품ID = PURCHASE.상품ID
```

> \* 은 '모든 것'을 나타냄

example 1 from p.14

```SQL
SELECT 고객ID, SUM(구매금액)
FROM PURCHASE
GROUP BY 고객ID
```

> 모든 계산 공식은 SELECT 안에 들어감
> SELECT 안에 AGGREGATE FUNCTION이 있다면 GROUP BY가 꼭 필요함 (연산의 대상을 지칭해줘야); vice versa

example 1 from p.15

```SQL
SELECT DeliveryMethod, PayMethod
FROM PURCHASE
ORDER BY DeliveryMethod, PayMethod
```

> ORDER BY의 경우, 앞에 있는 것으로 먼저 정렬 후 다음 것으로 정렬

example 2 from p.16

```SQL
SELECT CUSTOMER.고객이름,
    COUNT(1) AS 구매횟수,
    SUM(구매금액) AS 구매금액총합,
    SUM(구매금액)/COUNT(1) AS 평균구매금액
FROM CUSTOMER, PURCHASE
WHERE CUSTOMER.고객ID = PURCHASE.고객ID
GROUP BY CUSTOMER.고객이름
```

> COUNT(1) primary key를 기준으로 센다.

example 3 from p.16

```SQL
SELECT CUSTOMER.성별,
    COUNT(1) AS 구매횟수,
    SUM(구매금액) AS 구매금액총합,
    SUM(구매금액)/COUNT(1) AS 평균구매금액
FROM CUSTOMER, PURCHASE
WHERE CUSTOMER.고객ID = PURCHASE.고객ID
```
